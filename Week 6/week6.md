## Chart 1: Which star properties are most useful for telling star types apart?

**File:** `Week 6/charts_static/q1_type_separation.png`

The chart is a heatmap of star type by property. The properties are temperature, log10 luminosity, log10 radius, and absolute magnitude. I standardized the mean values for each property into z-scores so they can be compared on the same color scale. That matters because temperature, luminosity, radius, and magnitude do not use the same units.

**Why this chart type:** The question asks which properties create the clearest contrast across categories. A heatmap shows all six star types and four properties in one view without the label overlap that ruined the faceted box plot export. Z-scores let the four properties share one color scale even though they use different units.

**How to read it:** Each cell is a z-score for that star type's mean on that property. Strong color contrast across rows means that property separates types well. Values near 0 are close to the dataset-wide average. The colorbar is labeled "Z-score"; the RdBu colorscale is centered at 0 (`zmid=0`).

This chart answers the first question by showing which properties create the clearest contrast across star types. If one property has strong color changes from one star type to another, that property is doing more work in separating the categories.

The main pattern I see is that luminosity and radius separate the types especially well. Supergiants and hypergiants are high on log luminosity, while brown dwarfs, red dwarfs, and white dwarfs are low. Radius also separates the giant classes from the dwarf classes, with hypergiants standing out the most. Temperature helps, but it is not as clean by itself. White dwarfs are hot but small and dim, so temperature alone would not tell the whole story.

Absolute magnitude also helps, but it has the astronomy-specific twist that lower or more negative values mean brighter stars. That is why the giant classes have low absolute magnitude values while dwarfs have higher values. Overall, I would infer that star type is best explained by using multiple properties together, especially luminosity and radius, instead of relying on one column.

**Code behind it:** `question1_profile_heatmap()` in `week6_stars_visuals.py` — `groupby` means, z-score standardization, `go.Heatmap` with numeric labels on cells. Written justification in `chart_justifications.md` item 1.

## Chart 2: Are larger stars always brighter, or does the relationship change by star type?

**File:** `Week 6/charts_static/q2_radius_luminosity.png`

This chart is a scatter plot with radius on the x-axis and luminosity on the y-axis. Both axes use log scale because radius and luminosity vary by many orders of magnitude. Each point is one star, and the color shows star type.

**Why this chart type:** The question is about the relationship between two numeric variables. A scatter plot lets me see whether the points move upward as they move to the right. Log-log axes are appropriate because without log scale, extreme giant stars dominate and correlation on raw values (`r ≈ 0.527`) understates the pattern visible in log space (`r ≈ 0.918`).

**How to read it:** Right means larger radius; up means higher luminosity. Color shows star type. Point size encodes temperature in the interactive HTML version; the static PNG keeps color as the main type encoding.

This is the right chart type for the question because the question is about the relationship between two numeric variables. A scatter plot lets me see whether the points move upward as they move to the right. If they do, larger stars tend to be brighter.

The plot does show a strong upward pattern overall. When I calculated the correlation on the log-transformed radius and luminosity values, it was about `0.918`, which is very strong. On the raw values the correlation was lower, around `0.527`, because the extreme giant stars dominate the scale. That difference is exactly why the log-log view is useful.

The answer is not simply "bigger always means brighter." The relationship is strong overall, but the points cluster by star type. White dwarfs are a good example of why the simple rule breaks down: they can be hot, but they are small and not very luminous compared with giant stars. The chart supports a more careful answer: larger stars are often brighter in this dataset, but star type changes the relationship.

**Code behind it:** `question2_radius_vs_luminosity()` — `px.scatter` with `log_x=True`, `log_y=True`, `STAR_TYPE_COLOR_MAP`, hover data for temperature, magnitude, cleaned color, and spectral class. Written justification in `chart_justifications.md` item 2.

## Chart 3: Does a star's color reliably indicate its temperature and type?

**File:** `Week 6/charts_static/q3_color_temperature.png`

This chart is a box plot of temperature by cleaned star color, with star type shown by color. I cleaned the `Star color` labels because the CSV had repeated versions of the same idea, such as `Blue-white`, `Blue White`, and `Blue white`. Without that cleanup, the x-axis would split one category into several fake categories.

**Why this chart type:** The question is whether color groups have distinct temperature distributions. Box plots show median, spread, and overlap between groups. Categories on the x-axis are sorted by mean temperature (cool to hot) via `groupby("Star_color_clean")["Temperature (K)"].mean().sort_values()`.

**How to read it:** Each box is a cleaned color category. Higher boxes mean hotter stars on average. Overlap between boxes means color alone would misclassify some stars. Star type color within each box shows whether a color label maps to one type or many.

This chart answers the third question by showing whether color groups have distinct temperature ranges. If color were a reliable predictor, the boxes would be separated with little overlap.

The pattern mostly follows the basic astronomy expectation: red stars are cooler on average, while blue and blue-white stars are hotter on average. In the cleaned data, red stars have a mean temperature around `3292 K`, while blue stars average around `21918 K`, and blue-white stars average around `16660 K`.

But the chart also shows overlap and exceptions. Some color categories have wide ranges, and some categories only have a few rows. That makes color a useful clue, not a perfect classifier. I would infer that color can help estimate temperature directionally, especially red vs blue, but I would not use color alone to determine star type. It works better when combined with temperature, luminosity, radius, and spectral class.

**Code behind it:** `question3_color_temperature_type()` — `clean_star_color()` preprocessing, sorted `color_order`, `px.box` with `STAR_TYPE_COLOR_MAP`. Written justification in `chart_justifications.md` item 3.

### C3: Data cleaning and file handling

The charts read real CSV data from disk, not hardcoded numbers. `load_and_prepare()` in `week6_stars_visuals.py` calls `pd.read_csv(DATA_PATH)` where `DATA_PATH` points to `Week 5/stars.csv`. That path is built with `Path(__file__).resolve().parents[1] / "Week 5" / "stars.csv"` so the script works regardless of where it is run from.

Three messy data problems showed up immediately and had to be fixed before plotting:

1. **Numeric star type codes.** The CSV stores star type as integers 0–5. I map those to readable labels with `STAR_TYPE_LABELS` and `df["Star_type_label"] = df["Star type"].map(STAR_TYPE_LABELS)`.
2. **Inconsistent color spellings.** The `Star color` column had multiple spellings for the same category (`Blue-white`, `Blue White`, `Blue white`). `clean_star_color()` lowercases, strips whitespace, normalizes underscores, and looks up `COLOR_NORMALIZATION`. Unmapped values fall back to `.title()` so uncommon colors stay visible instead of being dropped.
3. **Extreme numeric ranges.** Luminosity and radius span many orders of magnitude. I add `log10_Luminosity` and `log10_Radius` columns, using `.where(value > 0)` before `np.log10()` so zero or negative values become `pd.NA` instead of crashing the script.

**Evidence:**

- `load_and_prepare()` in `week6_stars_visuals.py`: `pd.read_csv`, label mapping, log10 columns.
- `clean_star_color()` and `COLOR_NORMALIZATION` dict (lines 53–79).
- `week6_mp1_full.ipynb` data profile cell: `df.isna().sum()`, `value_counts()` on cleaned colors.
- Q3 box plot uses `Star_color_clean`, not raw `Star color`.

**Real problem solved:** Without color normalization, Q3 would mislead the viewer into thinking there are more distinct color groups than the dataset actually supports. Without log transforms, Q1 and Q2 would compress most star types into unreadable clusters.

### C5: Data analysis with pandas

The charts are visual versions of pandas operations, not decorations on top of unrelated numbers. Each chart answers a specific question using explicit dataframe work. See the chart sections above for full interpretation; the operations are:

**Q1 — Which properties separate star types?**

- Operations: `groupby("Star_type_label")[metric_cols].mean()`, then z-score standardization `(means - means.mean()) / means.std(ddof=0)`.
- Result: luminosity and radius show the strongest contrast between dwarf classes and giant/hypergiant classes.
- Interpretation: star type is best explained by multiple properties together, especially luminosity and radius.

**Q2 — Are larger stars always brighter?**

- Operations: `df["Radius(R/Ro)"].corr(df["Luminosity(L/Lo)"])` on raw values; `df["log10_Radius"].corr(df["log10_Luminosity"])` on log-transformed values (computed in `week6_mp1_full.ipynb`).
- Result: Pearson r on raw values is about `0.527`; on log10 values it is about `0.918`.
- Interpretation: strong overall positive relationship in log space, but clusters by star type show the simple rule breaks down for cases like white dwarfs.

**Q3 — Does color reliably indicate temperature and type?**

- Operations: `groupby("Star_color_clean")["Temperature (K)"].mean().sort_values()` to order x-axis categories cool-to-hot; box plot to inspect spread and overlap.
- Result: red stars average around `3292 K`, blue stars around `21918 K`, blue-white around `16660 K` — directionally correct, but boxes overlap.
- Interpretation: color is a useful clue, especially red vs blue, but not a perfect classifier for type.

**Evidence:**

- `question1_profile_heatmap()`: `groupby`, mean, z-score math.
- `question2_radius_vs_luminosity()`: `corr()` call on line 486; log-log scatter in final PNG.
- `question3_color_temperature_type()`: `groupby` for `color_order`, box plot on cleaned labels.
- `week6_mp1_full.ipynb`: prints Pearson r (raw vs log10) and markdown observations under each chart.
